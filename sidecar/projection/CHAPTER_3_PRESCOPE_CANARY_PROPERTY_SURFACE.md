# Chapter 3.4 pre-scope — Canary closure as property-test surface

## §1 Scope and load-bearing claim

Chapter 3.4 ships the **canary loop as a property-test surface**, not as hand-curated integration tests. The architectural commitment in `/home/user/outsystems-ddl-exporter/sidecar/projection/VISION.md:96-108` is concrete: FsCheck generators feed a tiered predicate library; `[<Property>]`-attributed tests replace the `Assert.Contains` discipline that currently rides at `JsonEmitterTests.fs:96-105` and `RichProfilingEndToEndTests.fs:280`. Per Appendix E §E.6 (`VISION_REVIEW.md:686-703`), the surface is **~600 lines of test code + ~200 lines of C# DacFx wrapper**, replacing the ~2000+ lines of curated DacpacEmitter integration tests that the deferred chapter-3 §5 nine-slice plan would otherwise demand.

The load-bearing claim has three parts:

1. **Tier-1 pure properties** (T1 byte/model equal, T11 sibling-chorus, T2 coproduct, A18 policy-orthogonality) cover ~80% of the axiomatic surface with **zero Docker** and sub-second feedback. They run in pre-commit.
2. **Tier-2 container-pooled deploy properties** (`roundTripBySsKey`, `idempotentRedeploy`, `renameSurvives`, `policyOrthogonal-on-deploy`) verify the actual deployment-side claims that T1 byte-determinism alone cannot reach. Per R2 (`VISION_REVIEW.md:98-108`), `idempotentRedeploy` is the explicit composition that turns the previously-implicit CDC-safety claim into an observable property: deploy → redeploy same DACPAC → assert `DacServices.GenerateDeployScript` returns empty.
3. **Tier-3 regression captures** are append-only `[<Theory>][<MemberData>]` tests fed by the shrunk failures from tiers 1–2.

The load-bearing claim **of the chapter** — what makes it the *primary* verification vehicle — is that every acceptance criterion in `VISION.md:110-120` (verifiable correctness, CDC safety, identity, T11 structural, V1 sunset gate) gets a typed predicate that names it. A failing property points at a specific axiom; a green property *is* the canary signal.

This chapter sits **after** chapter 3.3 (DacpacEmitter; the binary produced is the input to tier 2) and **partially before** the `ArtifactByKind` refactor (Appendix H §H.7 `VISION_REVIEW.md:1036-1049`). Tier-1 properties that depend on `ArtifactByKind` (e.g., `siblingChorusAgrees` as a one-line `Set.equal Map.keys`) ship on whichever side of the refactor the dependency lands; tier-1 properties that don't (`t1ByteEqual`, `coproductPreservation`) ship immediately.

### `genSsKey` — interned pool with stratification

Per the post-`ArtifactByKind` plan (`VISION_REVIEW.md:987-1010`), `SsKey` becomes the four-variant DU: `OssysOriginal of Guid | Synthesized of source*basis | DerivedFrom of parent*reason | V1Mapped of v1*ns`. The generator stratifies:

```fsharp
let genOssysGuid : Gen<Guid> =
    // Small interned pool — ~8 distinct guids across a generated catalog.
    // Collision rate matters: the more reuse, the more wireReferences
    // can produce cross-module FK pairs from the same generated set.
    Gen.elements [for _ in 1..8 -> Guid.NewGuid ()] |> Gen.resize 8

let genSsKey : Gen<SsKey> =
    Gen.frequency [
        7, genOssysGuid |> Gen.map SsKey.OssysOriginal      // honors A1 fully
        2, gen { let! src = Gen.elements ["json"; "rowsets"]
                 let! basis = Arb.generate<NonEmptyString>
                 return SsKey.Synthesized (src, basis.Get) } // documents the bound
        1, gen { let! parent = genOssysGuid |> Gen.map SsKey.OssysOriginal
                 return SsKey.DerivedFrom (parent, "inverse") } ]
```

The 7:2:1 stratification matches the cutover reality: most catalogs come through the rowsets path, a minority through the JSON path, and `DerivedFrom` only appears post-symmetric-closure. `V1Mapped` is *not* generated by `genSsKey`; it appears only in `RefactorLogEmitter` properties (chapter 3.5) where the cross-version threading is the test subject.

Until `SnapshotRowsets` lands and the `SsKey` DU is split, **the generator falls back to today's `Original of string` + a fresh `Guid.ToString()` body**. The CHAPTER_3.4 slice that flips the generator to the four-variant form is gated on chapter-5's DU refactor (Appendix H §H.7 step 5).

### `genName`, `genPrim`, `genAction`

```fsharp
let genName : Gen<Name> =
    // Reject blank input via Name.create's smart constructor; FsCheck's
    // NonEmptyString admits whitespace, which Name.create rightly rejects
    // (mirrors Fixtures.fs:28). Keep names short for shrinker readability.
    Arb.generate<NonEmptyString>
    |> Gen.suchThat (fun s -> not (System.String.IsNullOrWhiteSpace s.Get))
    |> Gen.map (fun s -> Name.create (s.Get.Substring(0, min 12 s.Get.Length))
                         |> Result.value)

let genPrim : Gen<PrimitiveType> =
    Gen.elements [Integer; Decimal; Text; Boolean; DateTime; Date; Guid; Binary]
let genAction : Gen<ReferenceAction> =
    Gen.elements [NoAction; Cascade; SetNull; Restrict]
```

`Time` is omitted from `genPrim` until the DacFx type-correspondence map names it (impedance per `CHAPTER_3_PRESCOPE_DACPAC_EMITTER.md` §2). Adding it later is a one-line change.

### `genAttrRaw` and `genKind`

Single source of truth on the at-least-one-PK invariant: enforce at `genKind`, not at `genAttrRaw`. Per `Catalog.fs:119-126`:

```fsharp
let genAttrRaw : Gen<Attribute> = gen {
    let! key  = genSsKey
    let! n    = genName
    let! pt   = genPrim
    let! col  = gen { let! cn = genName
                      let! nullable = Arb.generate<bool>
                      return { ColumnName = Name.value cn; IsNullable = nullable } }
    let! mand = Arb.generate<bool>
    return { SsKey = key; Name = n; Type = pt
             Column = col; IsPrimaryKey = false; IsMandatory = mand } }

let genKind (kindKey: SsKey) : Gen<Kind> = gen {
    let! attrCount = Gen.choose (1, 6)
    let! attrs     = Gen.listOfLength attrCount genAttrRaw
    // Invariant: exactly one PK on the first attribute (DacFx requires it;
    // composite-PK shapes are introduced via genCompositePk below).
    let attrs'     = attrs |> List.mapi (fun i a -> { a with IsPrimaryKey = (i = 0) })
    let! idxCount  = Gen.choose (0, 3)
    let! indexes   = Gen.listOfLength idxCount (genIndexFor attrs')
    let! schema    = Gen.elements ["dbo"; "sales"; "ext"]
    let! tbl       = genName |> Gen.map Name.value
    return { SsKey = kindKey; Name = ...; Origin = OsNative; Modality = []
             Physical = { Schema = schema; Table = tbl }
             Attributes = attrs'; Indexes = indexes; References = [] } }
```

Composite-PK kinds are a **separate generator** (`genCompositeKeyKind`) consumed at low frequency from `genModule` so the at-least-one-PK rule never breaks under shrinking.

### `genIndexFor attrs'`

```fsharp
let genIndexFor (attrs: Attribute list) : Gen<Index> = gen {
    let attrKeys = attrs |> List.map (fun a -> a.SsKey)
    let! cols   = Gen.choose (1, min 3 attrs.Length)
                  |> Gen.bind (fun n -> Gen.shuffle attrKeys |> Gen.map (Array.take n >> Array.toList))
    let! unique = Arb.generate<bool>
    let! n      = genName
    return { SsKey = ssKey ($"OS_IDX_{Guid.NewGuid():N}".Substring(0, 12))
             Name = n; Columns = cols; IsUnique = unique; IsPrimaryKey = false } }
```

`IsPrimaryKey = false` always — the kind's PK is a structural attribute flag, not a separate index entry. Composite indexes (`Columns.Length >= 2`) and unique-index variants are present by construction in this generator. **Filtered indexes are deferred** to a later slice (DacFx supports them; V2's IR has no `WhereClause` field yet — it would be IR-grows-under-evidence).

### `genModule` — fresh attribute SsKey scoping

```fsharp
let genModule (moduleKey: SsKey) (kindKeys: SsKey list) : Gen<Module> = gen {
    // Each call to genKind generates fresh attribute SsKeys via genAttrRaw;
    // no need to thread a context. The kind's own SsKey is supplied by the
    // caller so genCatalog can preallocate the unique kind-key set.
    let! kinds = kindKeys |> List.map genKind |> Gen.sequence
    let! n     = genName
    return { SsKey = moduleKey; Name = n; Kinds = kinds } }
```

### `wireReferences` — the load-bearing reference threader

```fsharp
type RefSpec = { SourceKindKey: SsKey; SourceAttr: SsKey; TargetKind: SsKey; OnDelete: ReferenceAction }

let wireReferences (allKinds: Kind list) : Gen<RefSpec list> = gen {
    if allKinds.Length < 2 then return [] else
    // Build the choice space first. For every (sourceKind, nonPkAttr) pair
    // and every (targetKind != sourceKind), a candidate RefSpec exists.
    let candidates =
        [ for src in allKinds do
            for srcAttr in src.Attributes do
              if not srcAttr.IsPrimaryKey then
                for tgt in allKinds do
                  if tgt.SsKey <> src.SsKey then
                    yield { SourceKindKey = src.SsKey
                            SourceAttr    = srcAttr.SsKey
                            TargetKind    = tgt.SsKey
                            OnDelete      = NoAction } ]
    let! refCount = Gen.choose (0, min 5 candidates.Length)
    let! shuffled = Gen.shuffle candidates
    let! actions  = Gen.listOfLength refCount genAction
    return shuffled |> Array.take refCount |> Array.toList
                    |> List.mapi (fun i r -> { r with OnDelete = actions.[i] }) }

let applyReferences (refs: RefSpec list) (modules: Module list) : Catalog =
    // Threads each RefSpec's ssKey-rooted Reference back into its source kind.
    ...
```

Cross-module FKs fall out for free — `tgt` is drawn from `allKinds` regardless of which module owns it. Cycles arise organically when two kinds reference each other; an opt-in `genCycleSpec` knob can bias the candidate filter to prefer back-edges (chapter 3.4 ships without it; ChapterCycleResolution properties live in `CycleResolutionTests.fs` already). The wiring step is **bottom-up by construction** — every FK target exists before a `RefSpec` for it is generated. No `Gen.suchThat` filtering, no shrinker collapse.

### `genCatalog` and size knobs

```fsharp
let genCatalog : Gen<Catalog> = gen {
    let! moduleCount = Gen.choose (1, 3)
    let! kindCounts  = Gen.listOfLength moduleCount (Gen.choose (1, 4))
    let modKeys      = [for _ in 1..moduleCount -> ...] |> List.map (fun g -> SsKey.OssysOriginal g)
    let kindKeysPerModule =
        kindCounts |> List.map (fun n -> [for _ in 1..n -> ...])
    let! bareModules =
        List.zip modKeys kindKeysPerModule
        |> List.map (fun (mk, kks) -> genModule mk kks)
        |> Gen.sequence
    let allKinds = bareModules |> List.collect (fun m -> m.Kinds)
    let! refs    = wireReferences allKinds
    return applyReferences refs { Modules = bareModules } }
```

**Size knobs** (per Appendix E §E.4 perf budget):
- Modules: `1..3` (cap at 3 — tier-2 deploy must finish in ~150 ms).
- Kinds-per-module: `1..4` (caps total kinds at 12).
- Attributes-per-kind: `1..6`.
- Indexes-per-kind: `0..3`, with `1..min 3 attrs.Length` columns each.
- References: `0..min 5 candidates.Length`.

These are deliberately small. The maintainer's instinct will be to widen them; resist until tier-2 budget allows.

## §3 Shrinking strategy

Per Appendix E §E.2 (`VISION_REVIEW.md:613-630`) — outermost first; biggest-blast-radius reductions before fine-grained ones.

```fsharp
let private dropOneModule (c: Catalog) : Catalog seq =
    seq { for i in 0 .. c.Modules.Length - 1 ->
            { c with Modules = c.Modules |> List.mapi (fun j m -> j, m)
                                          |> List.filter (fst >> (<>) i)
                                          |> List.map snd } }
let private dropOneKindFromAnyModule (c: Catalog) : Catalog seq = ...
let private dropOneReferenceFromAnyKind (c: Catalog) : Catalog seq = ...
let private dropOneIndexFromAnyKind (c: Catalog) : Catalog seq = ...
let private dropOneNonPkAttribute (c: Catalog) : Catalog seq = ...
let private shrinkOnePrimitiveType (c: Catalog) : Catalog seq =
    // Decimal -> Integer, Text -> Boolean, DateTime -> Date, Date -> Integer.
    // Never shrink Guid (identity-relevant); never shrink Binary (no smaller form).
    ...

let shrinkCatalog (c: Catalog) : Catalog seq =
    seq {
        yield! dropOneModule c
        yield! dropOneKindFromAnyModule c
        yield! dropOneReferenceFromAnyKind c
        yield! dropOneIndexFromAnyKind c
        yield! dropOneNonPkAttribute c
        yield! shrinkOnePrimitiveType c }
```

**Invariants the shrinker must preserve:**

1. **At-least-one-PK** — `dropOneNonPkAttribute` filters out the PK attribute by construction; never produces a kind with zero PKs. DacFx would reject the deploy and the tier-2 properties would noise on shrink.
2. **FK targets resolve** — `dropOneKindFromAnyModule` must also drop every `Reference` whose `TargetKind` matches the dropped kind. The shrinker cleans up referential integrity in the same step.
3. **No `SsKey` mutation** — identity is the spine (A4); mutating it invalidates every property that depends on identity-rooted matching. The shrinker only *removes* nodes; it never rewrites a `SsKey`.
4. **Module cohesion** — the last module is never dropped (catalogs are non-empty by `Catalog.fs:193-195`); every catalog has ≥1 module.

**Why FsCheck's default record shrinker is wrong:** the default would shrink `Name` character-by-character, producing nonsense like `Name "C"` and breaking the `Name.create` non-blank invariant; it would also independently shrink each field of `Catalog`/`Module`/`Kind`, producing referentially-incoherent intermediates (an `Index.Columns` list pointing at attribute SsKeys whose attributes were independently dropped). The custom shrinker preserves cross-field coherence by only ever removing whole-node units.

### `roundTripBySsKey` and `Catalog.equalModuloDacpacErasure`

The Π-erased axes are the named impedance points from `CHAPTER_3_PRESCOPE_DACPAC_EMITTER.md` §2:

```fsharp
[<RequireQualifiedAccess>]
module Catalog =
    /// Catalog equivalence under the axes DacFx cannot preserve through
    /// round-trip. Π-erased axes (Appendix E §E.5 — candidate A35 / T12):
    ///   Origin            — V2-IR-only tag; DacFx has no peer.
    ///   Modality          — V2-IR-only; DacpacEmitter writes as comments only.
    ///   Static populations — schema dacpac is data-free; populations belong
    ///                        to StaticSeedsEmitter, not the dacpac round-trip.
    ///   Lineage trail     — A26 — does not affect structural equality.

        let stripModality (k: Kind) = { k with Modality = []; Origin = OsNative }
        let stripCatalog (c: Catalog) =
            { c with Modules = c.Modules
                              |> List.map (fun m ->
                                   { m with Kinds = m.Kinds |> List.map stripModality }) }
        // Compare by SsKey-keyed sets, not list ordering — A33 deterministic
        // ordering applies to the schema artifact, not to the IR.
        let normalizeKindList ks = ks |> List.sortBy (fun k -> SsKey.rootOriginal k.SsKey)
        ...

let roundTripBySsKey (deploy: Catalog -> Result<Catalog, exn>) (c: Catalog) : bool =
    match deploy c with
    | Ok roundTripped -> Catalog.equalModuloDacpacErasure c roundTripped
```

### `idempotentRedeploy` — the CDC-safety predicate

```fsharp
let idempotentRedeploy
    (deploy: byte[] -> dbName -> Result<DeployArtifacts, exn>)
    (generateDeployScript: byte[] -> dbName -> Result<string, exn>)
    (emitDacpac: Catalog -> Result<byte[], exn>)
    (newDb: unit -> dbName)
    (c: Catalog)
    : bool =
    let dbName = newDb ()
    let bytes  = emitDacpac c |> Result.value
    deploy bytes dbName |> Result.value |> ignore   // first deploy lands all DDL
    match generateDeployScript bytes dbName with
    | Ok script ->
        // The DACPAC fast-lane assertion: the second deploy plan must be empty.
        // Only PRINT statements / comments are tolerable; any DDL noise fails.
        let normalized = script |> stripCommentsAndPrints
        System.String.IsNullOrWhiteSpace normalized
    | Error _ -> false
```

Key mechanism: `DacServices.GenerateDeployScript(DacPackage, dbName, ...)` returns the T-SQL the deployment *would* run, without actually running it (`CHAPTER_3_PRESCOPE_DACPAC_EMITTER.md:101-103`). On a model-equivalent redeploy, this script is empty (or contains only `PRINT 'Update complete.'` boilerplate). The CDC-record assertion (deploy → assert no rows in `cdc.dbo_<table>_CT`) is **chapter 4.1's** promoted-lane integration test, not chapter 3.4. Per VISION.md:154 — "The redeploy-zero-ALTER assertion lands here for the DACPAC fast lane; the SSDT/data-insert lane gets the same assertion in chapter 4.1's promoted-lane integration test."

### `renameSurvives` — A1 + DacFx incremental deploy verb inspection

```fsharp
let renameAttribute (c: Catalog) (attrKey: SsKey) (newName: Name) : Catalog =
    Catalog.mapAttribute attrKey (fun a -> { a with Name = newName }) c

let renameSurvives
    (deploy: byte[] -> dbName -> Result<unit, exn>)
    (generateDeployScript: byte[] -> dbName -> Result<string, exn>)
    (emitDacpac: Catalog -> Result<byte[], exn>)
    (newDb: unit -> dbName)
    (c: Catalog) (attrKey: SsKey) (newName: Name) : bool =
    let dbName = newDb ()
    deploy (emitDacpac c |> Result.value) dbName |> Result.value
    let renamed = renameAttribute c attrKey newName
    match generateDeployScript (emitDacpac renamed |> Result.value) dbName with
    | Ok script ->
        // DacFx's incremental deploy plan describes the verb. With a
        // RefactorLog entry communicating the rename, DacFx ALTERs.
        // Without one, DacFx DROP+CREATEs (the failure mode A1 names).
        not (script.Contains "DROP COLUMN") && script.Contains "ALTER COLUMN"
    | Error _ -> false
```

The `attrKey` and `newName` are extra `[<Property>]` parameters; FsCheck shrinks them alongside the catalog. The property's tier is **2** because it requires `deploy` and `generateDeployScript`. Until `RefactorLogEmitter` ships (chapter 3.5), this property fails trivially on every rename — that's the inversion VISION_REVIEW.md:680-682 names ("the property surface forces RefactorLogEmitter forward").

### `policyOrthogonal` — four-axis perturbation

The four axes (`Policy.fs:234-239`): Selection, Emission, Insertion, Tightening.

```fsharp
let perturbSelection (p: Policy) (c: Catalog) : Gen<Policy> = gen {
    let! kindKey = Gen.elements (Catalog.allKinds c |> List.map (fun k -> k.SsKey))
    return { p with Selection = ExcludeOnly (Set.singleton kindKey) } }

let perturbTightening (p: Policy) : Gen<Policy> = gen {
    let! enable = Arb.generate<bool>
    return { p with Tightening = if enable then someTighteningPolicy else TighteningPolicy.empty } }

// (perturbEmission and perturbInsertion symmetric)

let policyOrthogonal
    (axisPerturbation: Policy -> Catalog -> Gen<Policy>)
    (deploy: ProjectionInput -> Result<Catalog, exn>)
    (catalog: Catalog) (basePolicy: Policy) (perturbedPolicy: Policy) : bool =
    // Per A18 amended: emitters do not consume Policy. So Π output is
    // already Policy-invariant — what changes is the *Catalog reconstructed
    // after deploy*, which can vary if Selection filters kinds before emit.
    let baseDeployed = deploy { Catalog = catalog; Policy = basePolicy; Profile = Profile.empty } |> Result.value
    let pertDeployed = deploy { Catalog = catalog; Policy = perturbedPolicy; Profile = Profile.empty } |> Result.value
    Catalog.equalModuloAxis basePolicy perturbedPolicy baseDeployed pertDeployed
```

Four perturbation generators; four `[<Property>]` sites. Each property's pass criterion is "perturbing axis X changes only the deployed Catalog's axis-X projection; everything else is identical." This makes the V1 multi-environment-machinery assertion (`VISION_REVIEW.md:134-140`) into an observable predicate.

### `t1ByteEqual`, `t1ModelEqual`, `coproductPreservation` — pure, fastest

```fsharp
let t1ByteEqual (c: Catalog) : bool =
    let a = JsonEmitter.emit c
    let b = JsonEmitter.emit c
    a = b   // and likewise for RawTextEmitter; DacpacEmitter uses model equality

let t1ModelEqual (emitDacpac: Catalog -> byte[]) (c: Catalog) : bool =
    // Two emissions; load each into a TSqlModel; compare model graphs by
    // the impedance map's structural axes. No deploy; DacFx model API only.
    let bytesA = emitDacpac c
    let bytesB = emitDacpac c
    TSqlModel.structuralEqual (loadModel bytesA) (loadModel bytesB)

let coproductPreservation
    (split: Catalog -> Catalog * Catalog)
    (combine: Catalog * Catalog -> Catalog)
    (emit: Catalog -> string)
    (c: Catalog) : bool =
    let m1, m2 = split c
    let combined = combine (m1, m2)
    // T2: emit (m1 ⊕ m2) ≡ emit m1 ⊕ emit m2 modulo deterministic ordering.
    // The combine helper preserves module decomposition; emit on the
    // combined catalog must produce a serial concat of per-module emits.
    canonicalize (emit combined) = canonicalize (emit (combine (m1, m2)))
```

These three plus `siblingChorusAgrees` are the **fastest to ship** and the most-covered tier-1 properties; they go into the chapter's first slice.

## §6 DacFx wrapper for tier 2

C# (per `CHAPTER_3_PRESCOPE_DACPAC_EMITTER.md` §1 line 122 — F# calls the C# wrapper across a value-typed seam). Lives in **`Projection.Pipeline.DacFx`** (a new C# project; the canary's wrapper is the first DacFx integration in the codebase, and `Projection.Pipeline` is the named host per `DECISIONS 2026-05-15`). The seam:

```csharp
namespace Projection.Pipeline.DacFx;

public sealed class DacFxAdapter : IDisposable {
    private readonly string _connectionString;     // master connection
    private readonly DacServices _services;        // owns the SQL Server connection lifetime

    public DacFxAdapter(string connectionString) {
        _connectionString = connectionString;
        _services = new DacServices(connectionString);
    }

    /// Deploy a dacpac to a fresh database. Validation errors thrown by
    /// DacFx (DacServicesException) are caught and wrapped as Result.Error.
    public Result<DeployArtifacts, Exception> Deploy(byte[] dacpacBytes, string dbName) {
        try {
            using var stream = new MemoryStream(dacpacBytes);
            using var pkg = DacPackage.Load(stream);
            _services.Deploy(pkg, dbName, upgradeExisting: true, options: defaultDeployOpts);
            return Result.Ok(new DeployArtifacts(/* ... */));
        } catch (Exception ex) {
            return Result.Error<DeployArtifacts, Exception>(ex);
        }
    }

    /// Extract the deployed schema as a TSqlModel; the F# read-side adapter
    /// (chapter 3.1) translates back into Catalog. Tier-2 round-trip predicate
    /// uses this seam.
    public Result<Catalog, Exception> ExtractToCatalog(string dbName) { ... }

    /// Generate the deploy script that *would* be applied without actually
    /// applying it. The idempotentRedeploy predicate's mechanism.
    public Result<string, Exception> GenerateDeployScript(byte[] dacpacBytes, string dbName) {
        try {
            using var stream = new MemoryStream(dacpacBytes);
            using var pkg = DacPackage.Load(stream);
            return Result.Ok(_services.GenerateDeployScript(pkg, dbName, defaultDeployOpts));
        } catch (Exception ex) { return Result.Error<...>(ex); }
    }

    public void Dispose() { /* DacServices does not implement IDisposable;
                              connections are pooled by SqlClient. */ }
}
```

**Lifetimes:** `DacPackage` and `MemoryStream` are scoped to the call (`using`). `DacServices` is owned by `DacFxAdapter` — one per fixture lifetime; SQL connection pooling handles the per-database concurrency. The F# tests construct a `DacFxAdapter` once (in `SqlServerFixture.InitializeAsync`) and reuse it across cases; the `dbName` is the per-case salt.

**Exception-driven validation:** DacFx throws `DacServicesException` on validation errors (e.g., FK target missing, ambiguous column). The wrapper catches and converts to `Result.Error`. The F# property reads `Result.Error _ -> false`, which causes a property failure (counterexample shrinks). This is the right shape — generator bugs surface as shrinkable property failures, not as test-runner crashes.

## §8 Pre-emptive axiom amendment list

The chapter touches AXIOMS.md in three places. Drafts below; final wording is the chapter's responsibility.

### T1 amendment — composition for binary

Current T1 amendment (`AXIOMS.md:452-461`) stops at "same triple, same surface, bit-identical." Per R2 (`VISION_REVIEW.md:98-108`), the binary path requires composition with DacFx's idempotent-redeploy property:

> **T1 amended (2026-05-20) — binary normal form composes with DacFx idempotency.** For binary surfaces (DACPAC), determinism is content-equality under `TSqlModel.structuralEqual` (loaded model graphs), not byte-equality of the zip envelope. CDC-safety is the *composition* `T1_binary × DacFx.idempotentRedeploy`: V2 emits model-equivalent output; DacFx's diff planner against an existing schema produces zero ALTERs. The composition is the verifiable property; T1 alone is not the CDC claim.

### A35 candidate (or T12) — Π-erased axes are explicitly enumerated

Per Appendix E §E.5 (`VISION_REVIEW.md:683`):

> **A35 (new, candidate) — Π-erased axes are named.** Each binary surface declares which Catalog axes its round-trip cannot preserve. DacpacEmitter erases `Origin`, `Modality`, `Static.populations` (data, not schema), and `Lineage` (A26). The erasure surface is the `Catalog.equalModuloDacpacErasure` function; future binary surfaces declare their own equivalence relation. The function *is* the axiom — its body is the contract.

Naming as A35 (axiom) vs T12 (theorem): if `equalModuloDacpacErasure` is **definitional** (we choose what to compare), it's an axiom. If it's **derivable** from existing axioms (Π's signature determines erasure), it's a theorem. The chapter's first slice surfaces the right framing under empirical pressure.

### Rename-survives lemma — A1 type-stratified property

Per Appendix H §H.5 (`VISION_REVIEW.md:985-1011`):

> **A1 stratified (post-`SnapshotRowsets`) — identity-survives-rename holds on `OssysOriginal`; bounded on `Synthesized`; inherits on `DerivedFrom` and `V1Mapped`.** The four-variant `SsKey` DU makes the bound type-visible. Property tests claiming A1 accept only `OssysOriginal` (and the root of `DerivedFrom`); on `Synthesized`, the same test documents the bound (the property holds for `OssysOriginal`-typed kinds; `Synthesized`-typed kinds are explicitly excluded from the universal quantifier).

The amendment text mirrors the existing AXIOMS.md `:47-72` bound prose but encodes it at the type level.
