## Strategic-frame axes (per DECISIONS 2026-05-15 chapter-open shape)

1. **The registry is the canonical definition site, not a parallel enumeration.** Each pass module's primary public surface becomes `<PassName>.registered : RegisteredTransform<'In, 'Out>`; the `let run` function becomes private; consumers invoke `registered.Run`. **No parallel "register this pass" call exists anywhere.** F# top-level evaluation order builds the registry up from each module's `.registered`. The single-definition-site discipline is the structural commitment; bypassing the registry becomes impossible once `Compose.run` traverses it. Mirrors A18 amended (Π consumes Catalog × Profile by structural type, never Policy); A41 is the Pass-side bidirectional sibling — the dichotomy as type-witnessed contract, not one-sided discipline.

2. **Pillar 9 manifests in code through `Sites : TransformSite list`.** Each `RegisteredTransform` entry carries a list of intra-pass `TransformSite` values; each `Site` carries `Classification : DataIntent | OperatorIntent of OverlayAxis` + `Rationale : string` (the harvest-discipline analysis prose). Intra-pass fidelity holds: `TopologicalOrderPass.registered.Sites = [{ SortKahn; DataIntent; "Kahn ordering depends only on graph topology" }; { SelfLoopHandling; OperatorIntent OrderingPolicy; "self-loop disposition is operator-supplied via SelfLoopPolicy" }]` — one registry entry, two classified sub-sites. The Sites surface is what code review now has structural type to point at; the closed-DU empirical-test discipline carries over to classification widening (adding a new `OverlayAxis` variant lights up errors only at registration sites; semantic interpretation untouched).

3. **Strongly-typed at the per-pass site; erased at the flat-list boundary.** Each pass module's `.registered : RegisteredTransform<Catalog, Catalog>` (or `RegisteredTransform<RawRowSet, CatalogFragment>` for adapter rules; `RegisteredTransform<Catalog, ArtifactByKind<'element>>` for emitters) is typed at its declaration site; consumers (`Compose.run`, manifest emitter, audit traversal) invoke `registered.Run` against the typed export directly. The registry's flat collection `TransformRegistry.all : RegisteredTransform<obj, obj> list` exists only for enumeration (totality scan, manifest digest, completeness property tests); the boxing happens at the registry boundary, not at the consumer call site. No `obj`-shaped invocations live in the hot path.

4. **`Compose.run` becomes a traversal, not a hand-coded composition.** Today `Compose.run` is `read >>= project >>= write`; `project` hand-codes each pass invocation + each emitter call. After A.4.7, `Compose.run` iterates `TransformRegistry.allInStageOrder` and invokes each `registered.Run` at the appropriate stage seam. `Compose.runWithSkeleton` is the same traversal with a predicate filtering to `Classification = DataIntent` (and `Sites` containing only `DataIntent` entries; mixed-classification passes contribute only their `DataIntent` Sites; `OperatorIntent` Sites are skipped). The hand-coded composition retires; the registry becomes load-bearing.

5. **Full-sweep retroactive scope, not incremental.** Per Q6 in DECISIONS 2026-05-15 (late): A.4.7 ships the registry surface AND classifies AND refactors every existing transformation site in one chapter. Incremental adoption is structurally impossible — the skeleton-purity property fails for any unclassified pass that emits a `LineageEvent` (the test asserts every event has `Classification = DataIntent` under `Compose.runWithSkeleton`; an unregistered pass leaks unclassified events). The full-sweep blast radius is ~30 modules × ~30-60 LOC; mechanical-edits precedent from chapter A.0' applies (closed-DU empirical-test discipline at the consumer-site boundary — making `let run` private breaks only direct callers; reroute via `.registered.Run` and the compiler enumerates the worklist).
